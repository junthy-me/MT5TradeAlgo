## Context

当前 `mt5/P4PatternStrategy.mq5` 已经把相邻线段的整段极值校验封装为统一 helper，并在前置段 `Pre0P0`、历史骨架段 `P0P1/P1P2/P2P3` 以及实时触发段 `P3P4` 上分别调用。用户现在希望保留现有“按线段做整段极值校验”的口径，但把这些校验从硬编码全启用改成运行时按段开关，并明确 `P3P4` 这位只控制 `P3` 的整段极值检查，不控制 `P4`。

这个 change 与 `add-directional-pattern-trading` 有直接交叉：新参数必须兼容 `LONG_ONLY / SHORT_ONLY / BOTH` 三种方向模式，且同一组段开关要同时作用于多头和空头镜像检测。因此设计上不能回退到旧的多头专用表述，而应在方向感知的检测路径上继续增量修改。

## Goals / Non-Goals

**Goals:**

- 新增单一运行时参数 `InpRequiredSwingExtremaSegments_Pre0P0_P0P1_P1P2_P2P3_P3P4`
- 让 `Pre0P0`、`P0P1`、`P1P2`、`P2P3`、`P3P4` 五段的整段极值约束可独立启用或禁用
- 保持现有整段极值判定 helper 和方向镜像规则不变，只改变“是否应用该约束”
- 明确 `P3P4` 的配置位只约束 `P3`，不对 `P4` 施加整段极值要求
- 对非法配置执行显式失败，避免静默回退导致回测口径不透明

**Non-Goals:**

- 不把极值口径改成跨多段的“摆动窗口极值”
- 不修改 `CondA / CondB / CondC / CondF` 或 `P5/P6` 的结构定义
- 不引入新的形态点位、线段或额外方向模式
- 不为旧参数名提供兼容别名

## Decisions

### 1. 用固定顺序的字符串参数承载段级开关

新增输入：

- `InpRequiredSwingExtremaSegments_Pre0P0_P0P1_P1P2_P2P3_P3P4`

该参数使用固定顺序的 5 位逗号分隔布尔字符串：

- 第 1 位: `Pre0P0`
- 第 2 位: `P0P1`
- 第 3 位: `P1P2`
- 第 4 位: `P2P3`
- 第 5 位: `P3P4`

默认值采用 `"true,true,true,true,true"`。

原因：

- MQL5 对标量 `input string` 支持稳定，便于 `.ini` 和 EA 参数面板直接配置。
- 参数名中直接编码段顺序，可以减少位置映射错误。
- 维持单参数形式，避免引入 5 个独立 bool 带来的配置冗余。

备选方案：

- 使用 5 个独立 `input bool`。
  否决原因：可读性尚可，但参数列表会继续膨胀，且不利于在配置文件里做批量对比。
- 使用 bitmask 整数。
  否决原因：表达紧凑但可读性差，容易配错。

### 2. 解析规则采用严格失败而非宽松回退

策略在初始化时解析该字符串，并要求：

- 恰好包含 5 个逗号分隔项
- 每项只能是大小写不敏感的 `true` 或 `false`
- 允许值两侧出现空格

任一条件不满足时，策略初始化失败并输出明确错误。

原因：

- 该参数直接影响模式筛选口径，静默回退会让回测结论失真且难以追踪。
- 显式失败更符合 OpenSpec 对运行时契约的要求。

备选方案：

- 非法输入时自动回退到全 `true`。
  否决原因：用户通常不会意识到参数被忽略。

### 3. 前四段沿用“双端点校验”，`P3P4` 保持“只校验 P3”

启用某一段约束时：

- `Pre0P0`、`P0P1`、`P1P2`、`P2P3` 继续校验该线段两个端点是否达到按当前方向应有的整段极值
- `P3P4` 只校验 `P3` 是否达到 `P3->P4` 线段按当前方向应有的整段极值
- `P4` 永远不受此参数定义的整段极值约束

原因：

- 这与现有实现结构一致，改动局部且不会引入新的 `P4` 语义。
- `P4` 是实时入场触发价，不是历史骨架端点，把它纳入同类端点极值约束会改变触发定义。

备选方案：

- 让 `P3P4=true` 时同时要求 `P4` 也是该段另一端极值。
  否决原因：会把实时触发规则从“确认 `P3` 未被破坏”改成“确认 `P4` 自身成为端点极值”，超出本次范围。

### 4. 参数作用于方向感知检测，但不改变方向镜像规则

段级开关只决定“是否执行整段极值检查”，不改变当前方向化点位角色：

- 多头: `P0/P2/P5=low`、`P1/P3/P6=high`、`P4=ask`
- 空头: `P0/P2/P5=high`、`P1/P3/P6=low`、`P4=bid`

当某段开关为 `true` 时，检测器按当前方向应用对应的端点极值角色；为 `false` 时，跳过该段整段极值检查。

原因：

- 这样能与前一个 directional change 保持正交，这次 change 只增加可配置性，不重写方向逻辑。

备选方案：

- 为多头和空头定义不同的段开关参数。
  否决原因：没有用户需求，也会把参数复杂度翻倍。

## Risks / Trade-offs

- [该 change 与 directional change 修改同一批 spec 和代码路径] -> Mitigation: 在 design 和 specs 中明确本 change 以方向感知版本为基线，而不是回退到旧的多头专用行为。
- [字符串参数位置含义记错会导致筛选口径错误] -> Mitigation: 通过长参数名写死段顺序，并要求初始化日志输出解析后的 5 位结果。
- [`P3P4` 的非对称语义容易被误解为同时约束 `P4`] -> Mitigation: 在 spec、README 和错误日志里重复说明 `P3P4` 仅控制 `P3`。
- [关闭部分历史段极值后，命中率提高但形态质量下降] -> Mitigation: 这属于预期权衡，交由回测验证，不在运行时做额外隐式保护。

## Migration Plan

1. 更新 delta specs，明确新参数格式、默认值、非法配置处理，以及五段约束的逐段语义。
2. 在实现中新增字符串解析与缓存结构，并将现有 `Pre0P0`、`P0P1`、`P1P2`、`P2P3`、`P3P4` 校验点改为按配置短路。
3. 同步更新 README、示例 `.ini` 和日志输出，确保用户可直接观察五段解析结果。
4. 通过编译和定向回测验证全 `true` 维持现有行为、局部 `false` 能按预期放松指定线段。

## Open Questions

- 无。参数名、值格式、默认值和 `P3P4` 的非对称语义已经确认。
