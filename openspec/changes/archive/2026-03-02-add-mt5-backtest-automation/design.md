## Context

这轮 XAUUSD M15 参数微调暴露出两个实际问题。第一，MT5 在 macOS + Wine 环境下对 `/config` 文件路径、编码和 report 输出都比较脆弱，手工点 GUI 很难高频迭代。第二，单次回测结果主要落在 agent/terminal 日志里，如果没有统一抽取逻辑，就很难稳定比较不同参数组的收益、胜率和出场原因。

当前已经有两类成型产物：
- 启动器：把 repo 内 ini 转成 `UTF-16LE + BOM` 的运行时配置并用 Wine 启 MT5 tester
- 解析器：从 MT5 日志中定位指定 run，提取闭仓交易、净点数、近似收益美元和原因分布

## Goals / Non-Goals

**Goals:**
- 让 repo 内的 tester ini 可以直接驱动 MT5 回测，不再依赖手工 GUI 配置。
- 让一次回测至少产出稳定的结构化摘要，便于参数横向比较。
- 记录这轮探索中可复现的代表性参数配置，尤其是用户参数基线与最佳纯参数候选 `c12`。

**Non-Goals:**
- 不把 MT5 HTML report 落盘稳定化作为本次 change 的必达目标。
- 不把参数优化器做成通用网格搜索框架。
- 不改变 `P4PatternStrategy.mq5` 的任何交易行为。

## Decisions

### 用 shell 启动器处理 Wine/MT5 配置兼容，再由 Python 编排等待结果

`run_backtest.sh` 负责最接近平台细节的部分：写出 `UTF-16LE + BOM` 运行时 ini、放到 `/tmp/mt5run`、并用 `Z:\\tmp\\...` 路径启动 MT5。`backtest_runner.py` 只负责编排、等待和解析结果，避免把 Wine 启动细节和日志分析耦在一起。

备选方案：
- 全部用 Python 直接起 Wine：可以做，但调试成本更高，且 shell 更适合处理路径和编码转换。

### 以 MT5 日志作为权威结果源，而不是依赖 HTML report

这台机器上的 MT5 tester 在 Wine 环境里对 `Report=` 的行为并不稳定，HTML report 不能保证每次都落盘。agent/terminal 日志则稳定存在，因此本次自动化以日志为权威结果源，按 run 起止时间和输入参数精确匹配目标回测。

备选方案：
- 只解析 HTML report：在当前环境里可靠性不足。

### 只保留代表性回测配置，而不是把所有试验点都定义为仓库契约

本次 change 只把“可复现基线”纳入 change：用户 2025-02-28 参数基线，以及最佳纯参数候选 `c12` 的训练/月度验证配置。其它大量中间试验配置属于探索痕迹，不作为能力契约的一部分。

备选方案：
- 把所有 `c1..c15` 都写进 spec：噪音太大，也会把临时实验误升格为长期接口。

## Risks / Trade-offs

- [MT5 日志格式变动会破坏解析] → 解析器用明确的 run 起止和 inputs 匹配，降低误判；后续如有 MT5 升级再局部修正正则。
- [脚本路径依赖当前本机 Wine/MT5 安装位置] → 在设计上明确这是本仓库当前环境的本地自动化工具，而不是跨机器分发的通用 CLI。
- [只保存代表性配置会丢失部分探索轨迹] → 通过最终结论保留最佳组和用户基线，避免仓库里长期堆积临时试验点。
