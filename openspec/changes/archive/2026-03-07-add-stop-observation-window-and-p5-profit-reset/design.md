## Context

当前交易管理已经有三类关键运行时门控：

- 同一 `P4` bar 只允许首次成功开仓
- 共享骨架首次成功后阻止后续重复开仓
- `profit_target` 平仓后启动止盈观察窗口

这三者都作用于“是否允许新开仓”，而老持仓的止盈止损管理仍独立继续运行。新需求要求把这个节奏控制扩展到 `hard_stop` / `soft_stop`，同时在持仓首次形成合格 `P5/P6` 后，让弱止损激活与止盈重设绑定到同一个状态迁移事件上。

这里的复杂点有两类：

- 观察窗口不再只有一种来源，同一品种可能同时存在止盈窗口和止损窗口
- 当前 `UpdateSoftStopState()` 的行为是“一旦首次激活就停止继续扫描”；新规则要求在“首次激活时刻”对全部合格 `P5` 候选做一次最低价选择，再一次性冻结结果

## Goals / Non-Goals

**Goals:**

- 为 `hard_stop` 和 `soft_stop` 增加与止盈观察窗口平行的止损观察窗口
- 保持观察窗口只影响新开仓，不影响已有持仓继续按原规则止盈止损
- 在首次 `P5/P6` 激活时，同时设置 `soft_loss_price` 和 `P5` 锚定止盈价
- 明确 `selectedP5` 的一次性最低价选择规则，避免后续实现分歧
- 用新的运行时输入暴露止损观察窗口长度和 `P5` 锚定止盈系数

**Non-Goals:**

- 不改变 `P0-P4` 的检测逻辑，也不改变初始入场时的 `profit_price = P4 + InpProfitC * (a+b1+b2)` 推导
- 不把观察窗口扩展为按骨架、按 ticket 或按出场盈亏绝对值分类的冷却系统
- 不将 `P5` 锚定止盈改成可重复更新的 trailing 机制

## Decisions

### 1. 为 `symbol + timeframe` 分别记录止盈观察窗口和止损观察窗口

`profit_target` 仍然记录到现有的止盈观察窗口状态；`hard_stop` 和 `soft_stop` 新增独立的止损观察窗口状态。入场门控按并联处理，只要任意一个观察窗口仍有效，就阻止新的买单。

原因：

- 这保留了“观察窗口是品种级再入场节奏控制”的既有语义。
- 分开记录两种窗口，后续仍可分别调参或单独关闭，而不会把不同出场原因压扁成一套状态。

备选方案：

- 统一复用单一 `lastObservationExitBarTime`
  否决原因：实现更简单，但会丢失“止盈观察期”和“止损观察期”作为不同业务约束的可解释性和可调性。

### 2. 止损观察窗口的触发源只包含 `hard_stop` 和 `soft_stop`

只有由 EA 管理的持仓因 `hard_stop` 或 `soft_stop` 成功平仓后，策略才写入止损观察窗口起点；`profit_target` 继续只写入止盈观察窗口。

原因：

- 用户要的是“止损后观察窗口保持和止盈窗口一致”，不是把所有退出都归为同一类。
- 这样可以在日志和回测中清楚区分“盈利后冷却”和“止损后冷却”。

备选方案：

- 任何非 `profit_target` 平仓都触发止损观察窗口
  否决原因：会把手动平仓、非 EA 平仓等不属于策略风险控制的路径也纳入门控。

### 3. `P5` 锚定止盈与弱止损激活绑定为同一次状态迁移

当持仓首次出现合格 `P5/P6` 集合时，策略在同一个事件中同时完成：

- 选定 `selectedP5`
- 设置 `soft_loss_price = InpSoftLossC * selectedP5`
- 设置 `profit_price = selectedP5 + InpP5AnchoredProfitC * (a+b1+b2)`
- 将该持仓标记为已完成首次 `P5/P6` 激活

原因：

- 用户要求两者的设置时机一致
- 弱止损和第二阶段止盈都依赖相同的 `selectedP5`，绑定在一次迁移里能避免两个价位锚点不一致

备选方案：

- 只在弱止损激活后保留原止盈价
  否决原因：不满足新需求
- 把 `P5` 锚定止盈改成独立于弱止损的第二套后处理
  否决原因：会造成同一时段内两个后置规则基于不同条件触发，语义更复杂

### 4. `selectedP5` 在首次激活时从全部合格候选中取最低价，并在首次改写后冻结

实现上不再简单使用“`P4` 后全局最低点 + 其后最高点”的单一路径，而是在首次出现合格 `P5/P6` 集合时，从该时刻全部满足激活条件的 `P5` 候选中选取价格最低的 `P5`。完成首次价位重设后，该持仓不再因后续新的 `P5/P6` 组合继续改写弱止损和止盈价。

原因：

- 这同时满足“与第一止损位设置时机一致”和“多个合格 `P5` 中选最低价”的要求
- 冻结规则可以避免把第二阶段止盈做成持续追踪式 trailing 机制

备选方案：

- 首次出现任意一个合格 `P5/P6` 就立即使用该 `P5`
  否决原因：不能满足“多个候选时选最低价”的业务要求
- 后续持续下修为更低的新 `P5`
  否决原因：会把系统改成动态 trailing 状态机，超出本次需求

### 5. `P5` 锚定止盈系数使用独立输入，不复用 `InpProfitC`

新增独立参数，默认值 `0.7`，专门用于首次 `P5/P6` 激活后的止盈重设公式。

原因：

- 入场阶段止盈和 `P5` 锚定止盈属于两个不同阶段的利润目标
- 复用 `InpProfitC` 会把两套行为耦合到同一调参轴上，降低可控性

备选方案：

- 复用 `InpProfitC`
  否决原因：调大或调小初始止盈时会被动改变第二阶段止盈，不利于调参

## Risks / Trade-offs

- [双观察窗口并联后，再入场频率会进一步下降] → Mitigation: 为止损观察窗口提供独立参数，并允许设置为 `0` 显式关闭。
- [首次 `P5/P6` 激活后若当前价格已达到新止盈位，可能同一轮立即平仓] → Mitigation: 在设计和日志中明确这是预期行为，因为新止盈价在激活后立即生效。
- [从“单一路径最低 P5”切换为“首次激活时从合格候选中选最低 P5”会改变回测结果] → Mitigation: 在规格和实现日志中明确 `selectedP5` 的选择规则，并在验证中覆盖多个候选 `P5` 场景。
- [同一品种多持仓时，任一持仓的止盈或止损出场都会启动品种级观察窗口] → Mitigation: 明确文档语义是 `symbol + timeframe` 级再入场节奏控制，而不是 ticket 级冷却。

## Migration Plan

1. 更新 proposal 和 delta specs，明确两套观察窗口和 `P5` 锚定止盈的规格行为。
2. 在运行时输入中新增止损观察窗口长度参数和 `P5` 锚定止盈系数参数，并补充输入校验。
3. 扩展 `SymbolRuntimeState` 和持仓状态，使其支持双观察窗口和首次 `P5/P6` 激活后的冻结状态。
4. 重写 `P5/P6` 激活路径，使其在首次合格时完成“最低 `P5` 选择 + 弱止损设置 + 止盈重设”。
5. 通过编译与回测验证双窗口门控、首次激活冻结和即时止盈边界行为。

Rollback:

- 删除止损观察窗口状态和对应入场门控，恢复为仅 `profit_target` 后观察期。
- 恢复首次 `P5/P6` 激活只设置弱止损，不改写止盈价。
