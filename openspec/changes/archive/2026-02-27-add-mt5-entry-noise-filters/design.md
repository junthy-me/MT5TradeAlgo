## Context

现有 MT5 P4 模式策略在 CondA 到 CondF 通过后就可以进入下买单流程，但这个门槛仍可能放过振幅过小、离真实买价过近或主波段不够强的噪声形态。你新增的两个限制都发生在“准备提交买单”这一阶段，因此它们不应改变原有 P0-P4 模式识别结果，而应作为入场前的额外过滤层。

实现该变更时存在两个关键约束：
- 第一个过滤条件依赖“下单的买价”，这意味着判断基准应当是实际准备提交的买入价格，而不是历史 P4 参考价。
- 两个过滤条件必须像 CondA 到 CondF 一样成为硬性前置条件，任意一个失败都要阻止发单，并给出明确日志原因。

## Goals / Non-Goals

**Goals:**
- 为策略新增两条入场噪声过滤条件，并使其成为买单提交前的必过校验。
- 新增两个可配置参数，并定义清晰的默认值和计算口径。
- 在模式快照或交易决策结构中记录新增条件的计算值和通过状态。
- 在日志中输出噪声过滤条件的结果，方便识别被过滤掉的候选信号。

**Non-Goals:**
- 不改变原有 CondA 到 CondF 的计算方式，也不修改 P0-P6 形态定义。
- 不改变强止损、弱止损和止盈逻辑。
- 不引入新的订单类型、延迟确认机制或额外风控模块。

## Decisions

### 1. 将两条噪声条件定义为独立的新 Cond，并仅作用于入场阶段

新增两条条件，建议在实现中命名为 CondG 和 CondH：
- CondG: `(a / buy_price) >= NoiseFilter_aValueCompBuyPricePercent / 100.0`
- CondH: `max(b1, b2) >= NoiseFilter_maxBValueCompAValueProd * a`

原因：
- 这样可以与既有 CondA 到 CondF 保持统一表达方式，降低后续维护理解成本。
- 这些条件的目标是“阻止下单”，而不是“否定模式存在”，因此适合作为入场层过滤条件而不是前置模式识别条件。

备选方案：
- 直接把两个条件并入模式匹配定义。否决原因是其中 CondG 明确依赖实时买价，它与历史模式识别不是同一层面的判定。

### 2. 使用实际准备提交的买价计算 CondG

CondG 中的 `buy_price` 应采用发单前可获得的实际买入参考价，例如 MT5 市价买单提交前的当前 ask，而不是历史 P4 价格。

原因：
- 用户需求明确写的是“下单的买价”，而不是 P4 点位价。
- 使用实际准备成交的价格能够更准确地衡量 `a` 相对于当前入场成本的有效振幅。

备选方案：
- 使用 P4 参考价代替买价。否决原因是实盘偏离会使 CondG 失真，无法真实反映噪声过滤目标。

### 3. 保持 CondG 和 CondH 的结果与原始值可记录

交易决策结构应补充：
- `noiseFilterAValuePercent = a / buy_price * 100`
- `noiseFilterMaxBValue = max(b1, b2)`
- `condG`
- `condH`

原因：
- 不仅要知道是否通过，还要能在日志中看到具体数值和阈值，便于调参。

备选方案：
- 只记录通过或失败的布尔值。否决原因是缺少调参与复盘所需的解释性信息。

### 4. 失败时阻止发单并记录明确原因

在策略准备提交买单时，如果 CondG 或 CondH 任一失败，则不得发单，并在日志中记录：
- 当前买价
- `a`
- `a / buy_price * 100`
- `max(b1, b2)`
- CondG、CondH 的阈值和通过结果

原因：
- 噪声过滤最容易让用户误以为“策略失效”或“没有信号”，因此日志必须说明是过滤掉而不是没识别到模式。

备选方案：
- 静默跳过。否决原因是无法区分无信号与被过滤信号，影响调试。

## Risks / Trade-offs

- [CondG 对实时买价敏感，可能在不同市场状态下放大过滤力度] -> Mitigation: 明确参数单位为百分比并输出实际计算值，便于调参。
- [将过滤条件放在入场阶段会出现“模式已匹配但未下单”的情况] -> Mitigation: 在日志中区分“模式匹配成功”和“入场过滤失败”。
- [参数过严可能显著减少交易次数] -> Mitigation: 提供默认值并保留调参空间，先以日志验证过滤比例。

## Migration Plan

1. 在 EA 输入参数中新增两个噪声过滤配置项及默认值。
2. 在交易决策或模式快照结构中加入 CondG、CondH 及其原始计算值。
3. 在实际发买单前增加噪声过滤校验逻辑。
4. 在日志中输出过滤通过/失败详情。
5. 通过编译和 Strategy Tester 观察新增条件是否阻止了原本会入场的噪声信号。

Rollback:
- 删除新增参数和入场前过滤判断，恢复为仅依赖 CondA 到 CondF 的发单逻辑。

## Open Questions

- `NoiseFilter_aValueCompBuyPricePercent` 的默认值 `1` 在实现中应解释为 `1%`，而不是小数 `0.01`。本变更默认按“参数值即百分数”处理。
- 如果后续策略从市价单扩展到挂单，CondG 的 `buy_price` 应继续使用“订单准备提交时的目标买价”，而不是固定绑定当前 ask。
