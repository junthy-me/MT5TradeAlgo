## Context

当前性能优化变更 `optimize-mt5-pattern-search-performance` 已经把历史搜索与实时 `P4` 触发拆层，但当前实现仍然存在两个会改变结果的点：一是历史候选缓存使用固定上限并发生截断，二是实时评估采用“第一个命中即返回”的选择方式。这些做法虽然提升了速度，但不满足“允许用空间换时间，但绝对不能影响计算结果”的约束。

当前需要的不是进一步引入启发式候选压缩，而是将优化路线收敛为“精确优化”：
- 历史层只在新收盘 bar 出现时更新
- 历史候选集完整保留，不允许截断
- 只允许使用严格等价剪枝
- 最终候选选择必须与优化前全量搜索版保持一致

## Goals / Non-Goals

**Goals:**
- 保持历史层与实时层拆分带来的性能收益。
- 消除固定上限截断和近似返回对结果的影响。
- 允许增加内存占用，以换取结果精确性与运行时效率的平衡。
- 明确哪些剪枝属于严格等价剪枝，并要求最终结果与原始全量搜索实现完全一致。

**Non-Goals:**
- 不改变 P0-P6 点位定义、空间变量公式、时间变量公式或 CondA-H 语义。
- 不使用局部极值、zigzag、邻近候选合并、top-N 保留等可能改变候选集合的启发式方法。
- 不改变风控、噪声过滤、持仓管理或日志语义。

## Decisions

### 1. 历史候选缓存必须完整保留，不允许固定上限截断
历史层 SHALL 缓存完整候选集，而不是像当前实现那样最多保留固定数量的候选后丢弃其余结果。允许通过动态数组、分桶数组或索引化存储来增加内存使用，但不允许以任何形式丢弃合法候选。

原因：
- 候选截断直接改变参与实时评估的集合，结果不再与原始实现等价。
- 这违背了“绝对不能影响计算结果”的约束。

备选方案：
- 增大固定上限。否决原因是这只是推迟截断，不是消除截断。
- 只保留最近或最优候选。否决原因是会改变结果集合。

### 2. 实时层必须复现原始全量搜索的最终选择语义
实时 `P4` 评估不能采用“第一个命中即返回”的方式，而必须在完整历史候选集上完成评估后，按优化前全量搜索版的同等优先级规则选出最终候选。该选择语义至少需要复现“最新 `P3` 优先”以及原始枚举顺序在同优先级下的稳定性。

原因：
- 即使候选集完整，如果最终选择语义不同，输出结果仍可能不同。

备选方案：
- 继续使用第一个命中。否决原因是它是近似行为，不保证与全量搜索版一致。

### 3. 只允许严格等价剪枝
性能优化只能使用数学上可证明不会影响最终结果的剪枝，例如：
- 相邻跨度超过 `InpAdjustPointMaxSpanKNumber` 的组合
- 历史时长不满足 `CondE` 的组合
- 点位角色基础关系不满足的组合
- 基于 `CondA` 反推出的精确 `P0` 可行价格区间筛选

原因：
- 这些剪枝不会删除任何本可成为最终合法候选的组合。

备选方案：
- 使用局部极值候选提取。否决原因是它可能漏掉合法组合。
- 使用 zigzag 或波段阈值压缩。否决原因是它改变候选定义。

### 4. 历史缓存优先存储“骨架索引 + 历史变量”，而不是完整快照
为了用空间换时间但避免无意义的内存膨胀，历史缓存优先存储：
- `i0/i1/i2/i3`
- `P0/P1/P2/P3`
- `a/b1/b2/r2`
- `t1/t2/t3`
- `CondA/CondE/CondF` 的历史层结果
- 用于复现原始最终排序的顺序键

原因：
- 这比缓存完整 `PatternSnapshot` 更省空间。
- 实时层只需补 `P4/c/r1/t4/CondB/CondC/CondD` 即可完成评估。

备选方案：
- 缓存完整 `PatternSnapshot`。可行但内存冗余更大。

## Risks / Trade-offs

- [完整候选缓存可能占用较多内存] -> Mitigation: 使用索引化骨架存储，而不是完整快照存储。
- [若未精确定义原始全量搜索的最终选择规则，优化后仍可能结果漂移] -> Mitigation: 将原始实现的排序和 tie-break 规则显式写入验证标准。
- [严格等价剪枝实现复杂度更高] -> Mitigation: 先实现最直接的等价剪枝，再逐步加入 CondA 反推等更强剪枝。
- [验证成本高] -> Mitigation: 引入固定数据窗口和固定实时价格输入，对比 `P0-P4`、变量、条件和交易决策全量一致性。

## Migration Plan

1. 去除当前历史候选固定上限和截断逻辑。
2. 将历史缓存改为完整骨架缓存，并保留原始搜索顺序信息。
3. 将实时评估改为完整扫描并按原始全量搜索语义选最终候选。
4. 引入严格等价剪枝，减少无效组合计算但不改变结果。
5. 使用相同输入数据对比优化前全量实现与新实现，确认结果完全一致。

Rollback:
- 恢复为未经性能优化的原始全量搜索实现，作为精确性基准版本。

## Open Questions

- 原始全量搜索版在同一 `P3`、同一实时价格下的 tie-break 规则是否需要额外显式排序键，还是当前枚举顺序已足够作为规范。
- 完整历史候选缓存采用动态数组还是按 `P3` / `(P2,P3)` 分桶存储，更利于后续保持性能与可维护性。
