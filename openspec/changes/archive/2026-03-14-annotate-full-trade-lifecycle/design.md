## Context

当前 `P4PatternStrategy` 已经具备图表标注能力，但标注内容仍偏向“成功入场后的局部结构”，还没有把单笔交易完整生命周期涉及的点位、结构值和风控线稳定、统一地可视化出来。仓库同时已经支持多空镜像交易，因此新的标注设计需要以“交易生命周期”为中心，而不是继续沿用旧的“买入后补画”叙述。

这次 change 的关键约束有两个：
- 只允许改动 chart annotation 相关代码，不改模式匹配、下单、持仓管理或风控判定逻辑；
- 点位颜色要使用用户给定的固定映射，并且同名点位跨交易保持一致，不允许按 ticket 或方向动态换色。

## Goals / Non-Goals

**Goals:**
- 在当前策略的成交后可视化流程中，完整展示该交易生命周期内涉及的 `Pre0/P0/P1/P2/P3/P4/P5/P6` 点位。
- 在图表上补充 `a`、`b1`、`b2`、`c`、`d`、`e` 以及强止损线、弱止损线的文本或线条标注。
- 为 `Pre0`、`P0`、`P1`、`P2`、`P3`、`P4`、`P5`、`P6` 固定颜色映射，并保证每个点位颜色互不相同、跨交易保持一致。
- 保持现有对象命名空间与生命周期更新方式，避免覆盖其他交易的图形对象。

**Non-Goals:**
- 不调整模式识别、点位计算、入场/出场、止盈/止损或 `P5/P6` 激活逻辑。
- 不新增运行时参数，也不让颜色或是否显示结构值变成可配置项。
- 不改变日志契约，除非现有标注辅助日志必须附带新的 annotation 状态。

## Decisions

### 1. 复用现有分阶段标注流程，而不是重做单次全量重绘

实现将继续复用现有的两个主要标注时点：
- 成功入场后，绘制当下已知的 `Pre0/P0/P1/P2/P3/P4`、`a/b1/b2/c` 和强止损线；
- 首次 `P5/P6` 激活后，在同一交易对象命名空间内补充 `P5/P6`、`d/e` 和弱止损线。

这样做的原因是 `P5/P6`、`d/e` 和弱止损位只有在持仓后续演化中才成立，强行在入场时一次性“预绘制完整生命周期”会引入虚假的未来信息。替代方案是每次事件都销毁并重绘整套对象，但这会放大对象闪烁和命名管理复杂度，而且没有必要。

### 2. 使用单一颜色映射函数作为点位颜色契约

点位颜色将统一收敛到一个固定映射函数，例如：

```cpp
color GetAnnotationPointColor(const string pointLabel)
  {
   if(pointLabel == "Pre0")
      return(clrRed);
   if(pointLabel == "P0")
      return(clrYellow);
   if(pointLabel == "P1")
      return(clrOrangeRed);
   if(pointLabel == "P2")
      return(clrHotPink);
   if(pointLabel == "P3")
      return(clrMagenta);
   if(pointLabel == "P4")
      return(clrGold);
   if(pointLabel == "P5")
      return(clrWhite);
   if(pointLabel == "P6")
      return(clrKhaki);
   return(clrWhiteSmoke);
  }
```

这保证：
- 相同点位跨多笔交易使用同色；
- 颜色逻辑集中，后续扩展或核对时不需要分散查找多个绘图函数；
- 不会因为方向不同而改变点位颜色语义。

替代方案是让不同阶段或方向使用不同配色，但这会破坏“同名点位一眼可识别”的目标。

### 3. 结构值标注采用“按交易对象组追加文本”的方式

`a`、`b1`、`b2`、`c`、`d`、`e` 和止损线不会改变底层交易逻辑，只会作为图表对象追加到该交易自己的对象组里。文本对象和水平线对象继续依附当前的对象命名空间，并通过 ticket、`p4_bar_time` 等标识隔离不同交易。

这样做可以确保：
- 标注失败不会影响交易；
- 同一图表多笔交易可以并存；
- 现有的 annotation helper 可以局部扩展，而不是重构策略主流程。

### 4. 规范按方向中立表达结构值，而不是保留多头旧文案

虽然点位是镜像模式下共用的，但图表展示层应直接显示方向中立的 `a/b1/b2/c/d/e`，不再沿用历史上只偏向上涨场景的文本表述。结构值的数值来源仍复用当前方向抽象后的计算结果，因此空头不会引入单独一套公式。

## Risks / Trade-offs

- [图上对象数量继续增加，可能影响密集交易区可读性] → 继续复用按交易隔离的对象命名空间，并让文本/线条使用一致命名规则，便于后续按组清理。
- [补画 `P5/P6` 后文本堆叠，遮挡已有对象] → 保持已有阶段式追加机制，并尽量把结构值文本集中在单个或少量摘要对象上，而不是为每个值新开大量分散标签。
- [颜色在浅色/深色背景下可读性不一致] → 本次严格遵循用户给定配色，不引入背景自适应逻辑；必要时后续单独提 change 处理。
- [实现时误触交易主流程] → 只在 annotation helper、颜色映射和对象创建路径内修改，避免进入下单、止损判断和模式匹配代码。
