## Context

当前仓库已经包含产品需求文档和 OpenSpec 资产，但还没有现成的 MT5 Expert Advisor 源码。本次变更将新增一个 `.mq5` 策略，用于在 MetaTrader 5 中安全运行，识别 PRD 定义的 P0-P6 K 线形态，并以可配置风控管理实际买入持仓。

实现上存在几类约束：
- 模式识别是有状态的，既依赖相邻点之间的价格空间关系，也依赖时间跨度关系。
- 相邻点之间可能跨越多根 K 线，因此检测逻辑不能假设所有拐点都落在单根 bar 上。
- 策略会触发真实下单，因此交易行为必须具备确定性和可审计性。
- EA 需要支持多品种轮询，同时受到单品种持仓上限与轮询节奏的约束。

## Goals / Non-Goals

**Goals:**
- 设计一个检测器，能够从 MT5 bar 数据中提取候选 P0-P6 点位，并按配置校验 CondA 到 CondF。
- 定义交易管理行为，使策略在 P4 参考价位入场，并处理强止损、可选弱止损和止盈出场。
- 定义支持多品种、统一输入参数、结构化日志和后续扩展的运行时结构，避免后续重写检测逻辑。
- 通过职责拆分让策略保持可验证性，使计算、匹配和交易执行可以分别检查。

**Non-Goals:**
- 不包含做空逻辑，也不实现当前模式的空头镜像版本。
- 不包含自适应仓位控制、组合级风控或外部信号融合。
- 不构建除 MT5 日志和平台状态之外的回测框架、优化界面或持久化层。
- 不扩展到 PRD 之外的其他形态家族，仅覆盖当前单一的 P0-P6 结构。

## Decisions

### 1. 采用分层 EA 结构，显式拆分检测与交易管理模块

EA 将分成以下职责层：
- `input/config`：解析交易品种和各类阈值配置
- `market data`：读取 bar、当前价格和品种运行状态
- `pattern detector`：提取候选点位并校验 CondA-F
- `trade manager`：负责入场、止损、止盈和持仓上限控制
- `logging`：统一输出匹配结果和订单事件日志

原因：
- 将纯计算逻辑与 MT5 下单 API 隔离，能降低回归风险。
- 便于独立验证模式计算逻辑，而不必与交易副作用耦合测试。

备选方案：
- 使用一个巨大的 `OnTimer` 过程实现全部逻辑。否决原因是状态、计算和交易副作用混杂后难以安全审查。

### 2. 用标准化结构表达一次完整匹配结果

检测器将产出一个模式快照，至少包含：
- P0 到 P6 的点位时间和价格
- 空间变量 `a`、`b1`、`b2`、`c`、`d`、`e`、`r1`、`r2`、`sspanmin`
- 时间变量 `t1` 到 `t6`、`trigger_pattern_total_time_minute`、`tspanmin`
- CondA 到 CondF 的逐项通过状态
- 入场价、强止损、弱止损可激活状态和止盈位等衍生交易参数

原因：
- 用单一快照在检测与执行之间传递数据，可以避免重复计算。
- 日志可以直接打印同一份快照，保证记录与下单决策一致。

备选方案：
- 在交易执行阶段重新根据原始 bar 计算各类风控价位。否决原因是这会增加匹配结果与执行结果漂移的风险。

### 3. 通过有界局部极值窗口提取候选点位

对每个品种和周期，检测器都从最近已收盘 K 线中搜索候选拐点。相邻点之间允许跨越不超过 `AdjustPointMaxSpanKNumber` 根 K 线，点位价格使用 `PointValueTypeEnum` 在 `KMax`、`KMin`、`KAvg` 中选择对应方式计算。

原因：
- 这与 PRD 中“相邻点之间可能横跨多根 K 线”的规则一致。
- 相比无界历史扫描，这种方式可以把运行时复杂度控制在可配置范围内。

备选方案：
- 用逐 tick 方式生成点位。否决原因是 PRD 的形态定义建立在 K 线结构上，稳定的 bar 数据才是自然输入。

### 4. 仅对完全匹配的形态触发入场，并在品种级执行限流

EA 只有在 CondA 到 CondF 全部满足后才允许创建买单。在发单之前，运行时还会检查：
- 当前品种是否可交易
- 该品种由 EA 管理的持仓数量是否低于 `InpMaxPositionsPerSymbol`
- 订单请求是否使用了配置的手数、滑点、魔术号和备注

原因：
- 能避免过度交易，也能避免误伤手动单或其他 EA 的持仓。

备选方案：
- 无视当前仓位上限，先缓存所有匹配信号。否决原因是 PRD 明确要求单品种持仓受上限控制。

### 5. 将弱止损建模为条件触发的次级退出规则

强止损和止盈会在模式匹配完成后立即计算。弱止损在入场时保持未激活，只有后续走势形成 P5 和 P6，并满足：
- 已识别出 `S_P4P5` 和 `S_P5P6`
- `(e - d) >= n * c`

时才会启用。一旦启用，弱止损与强止损任一触发都可以按当前市场价平仓。

原因：
- 这与 PRD 中“弱止损为后续确认后再生效”的分阶段风控模型一致。

备选方案：
- 入场时直接设置弱止损。否决原因是这与 PRD 的触发条件定义不一致。

### 6. 以 `OnTimer` 作为主要扫描循环

EA 将按照 `InpTimerMillSec` 在 `OnTimer` 中扫描配置的交易品种，并在每次回调中顺序处理每个品种，同时在内存中保存各品种状态。

原因：
- 与 PRD 的运行要求一致，不依赖单一图表品种的 tick 到达频率。
- 对 tick 不均匀的多品种组合更稳妥。

备选方案：
- 将全部逻辑放在 `OnTick` 中。否决原因是多品种监控会受单一品种 tick 稀疏或集中影响，导致扫描不可靠。

## Risks / Trade-offs

- [噪声行情中点位选择可能产生误判] -> 通过只使用已收盘 bar、限制相邻点跨度并记录每次匹配快照来缓解。
- [基于定时器的多品种扫描可能错过极短暂的盘中波动] -> 通过保留可配置轮询间隔，并将模式确认建立在收盘 K 线结构而非瞬时 tick 上来缓解。
- [实盘中不一定能以精确 P4 价格成交] -> 将 P4 作为参考价计算衍生价位，实际发市场单时使用当前可成交价格并结合允许滑点。
- [弱止损激活会增加状态管理复杂度] -> 为每个持仓维护显式的弱止损状态，并只在满足 SetSoftLoss 条件时进行一次性状态迁移。
- [实盘交易中的回归缺陷成本高] -> 通过隔离计算逻辑、在下单前增加防御性检查，并在上线前用 Strategy Tester 覆盖关键场景来缓解。

## Migration Plan

1. 新增 EA 源文件及其支持性 include 文件，不改动无关运行资产。
2. 在 MetaEditor 中编译 EA，先消除所有编译错误和警告，再进入测试阶段。
3. 运行 Strategy Tester，覆盖单品种检测、多品种扫描、强止损、弱止损激活和止盈出场等关键场景。
4. 先部署到模拟环境，使用保守手数和详细日志观察行为。
5. 在审查匹配变量、订单生命周期和品种持仓限制日志后，再决定是否进入实盘。

Rollback:
- 从图表中移除或停用该 EA。
- 由于这是新增能力，回滚主要是运行层面的停用，而不是数据迁移。

## Open Questions

- PRD 要求按 P4 价位买入，但在 MT5 实盘执行中更可能以接近当前 ask 的市场价成交，而不是严格成交在历史 P4 打印价。实现时需要明确记录参考 P4 价与实际成交价。
- PRD 没有给出寻找 P0-P6 候选序列的具体历史回看深度。实现阶段需要给出一个有界默认窗口，必要时再将其开放为可调参数。
