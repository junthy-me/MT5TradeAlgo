## Context

当前策略在构建 `P0-P3` 历史骨架时会无条件执行 `Pre0-P0` 前置下跌检查，并把 `Pre0` 一起写入快照和图表标注。这种实现把“是否启用前置过滤”与“如何计算前置跌幅”耦合在一起，导致操作者无法在保留主结构规则的同时临时关闭 `Pre0`。另一方面，历史骨架已经对 `Pre0-P0`、`P0-P1`、`P1-P2`、`P2-P3` 做了端点极值校验，但实时触发段 `P3-P4` 仍只检查 `P4 < P3` 与 `CondB/CondC`，没有验证 `P3` 是否仍是该段最高点。

本次变更横跨检测规则、运行时输入和图表标注三处行为：前置条件公式要改，前置条件要可禁用，且 `P3-P4` 需要补上与历史骨架一致的端点极值约束，因此需要先把边界和状态语义写清楚。

## Goals / Non-Goals

**Goals:**
- 将 `Pre0-P0` 最小跌幅公式改成基于 `(a+b1)` 的门槛。
- 增加 `InpPreCondEnable`，并让其完全控制前置条件评估与 `Pre0` 标注是否出现。
- 在实时 `P4` 触发阶段强制 `P3` 达到 `P3-P4` 段的最高点，且默认允许并列极值。
- 保持现有历史骨架缓存、实时触发、图表标注命名空间和持仓管理流程不变。

**Non-Goals:**
- 不调整 `P0-P3` 历史骨架的候选排序规则。
- 不改变 `Pre0` 的搜索窗口、最小 bar 间隔和端点极值语义。
- 不修改 `P5/P6` 的 tick 顺序追踪与激活逻辑。

## Decisions

### 1. 将前置条件拆成“总开关 + 具体规则”两层
- 决定：新增 `InpPreCondEnable`，默认 `false`。当其为 `false` 时，`EvaluatePatternPreconditions(...)` 直接视为通过，并清空 `Pre0` 相关快照字段；当其为 `true` 时，继续执行 `EvaluatePriorDeclinePrecondition(...)`。
- 原因：用户要的是“只在显式启用时才做前置条件判断”，这不仅影响检测是否通过，也影响图表是否应出现 `Pre0`。让总开关先决于具体规则，可以同时约束识别和标注。
- 替代方案：保留当前始终启用，只把阈值设成 `0` 近似关闭。放弃原因是这仍会搜索和标注 `Pre0`，不符合需求。

### 2. 前置下跌最小跌幅只改结构基数，不改候选选择方式
- 决定：`minRequiredDrop` 改为 `InpPreCondPriorDeclineMinDropRatioOfStructure * (a + b1)`；`Pre0` 的搜索窗口、优先选择最高 `Pre0`、并列价格取更靠右者等现有策略保持不变。
- 原因：用户只修改跌幅基数，没有要求变更 `Pre0` 的挑选方式。保持候选选择规则不变，可以把影响收敛在阈值层。
- 替代方案：同时改 `Pre0` 选点优先级。放弃原因是这会引入额外行为变化，超出本次 scope。

### 3. `P3-P4` 极值校验放在实时触发阶段，按“同段允许并列极值”处理
- 决定：在 `EvaluateRealtimePatternFromBackbone(...)` 中新增 `P3-P4` 段校验，要求 `P3` 达到从 `P3` 所在 bar 到 `P4` 所在 bar/触发 tick 对应段的最高点；若段内存在与 `P3` 相同的 high，仍视为通过。
- 原因：`P4` 是实时 tick，不属于历史骨架缓存的一部分；把约束放在实时评估阶段，能复用现有缓存而不扩大历史缓存失效率。
- 替代方案：把 `P3-P4` 当成历史骨架的一部分预先缓存。放弃原因是 `P4` 只有在实时 tick 到来时才能确定，无法安全缓存。

### 4. `Pre0` 图表标注以快照字段是否有效为唯一判断条件
- 决定：图表层不单独读取 `InpPreCondEnable`，只根据 `pattern.preCondPriorDecline && pattern.pre0Time > 0` 决定是否绘制 `Pre0` 和 `pre0_drop` 标注。
- 原因：标注应该依赖模式快照，而不是重复读取运行时开关；这样可以保证回放、日志和图形都基于同一份快照事实。
- 替代方案：图表绘制函数内部再次读取 `InpPreCondEnable`。放弃原因是会造成绘图逻辑与快照状态分叉。

## Risks / Trade-offs

- [默认关闭前置条件会放宽默认匹配数量] → 通过 spec 和 README 明确说明默认值变化，并保留 `InpPreCondEnable=true` 的显式启用路径。
- [`P3-P4` 极值校验对实时 tick/bar 边界处理不清会导致误拒绝] → 实现时统一复用现有“端点达到整段极值、允许并列”的比较口径，只新增 `P3-P4` 专用 helper。
- [`Pre0` 关闭后旧图表对象仍可能残留] → 本次不做历史对象清理；仅保证新生成的模式在快照中未携带 `Pre0` 时不会创建对应对象。
