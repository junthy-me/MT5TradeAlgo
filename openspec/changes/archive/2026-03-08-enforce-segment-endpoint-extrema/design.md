## Context

当前历史骨架构造通过选择 `i0/i1/i2/i3` 四个端点，再用角色化取值规则得到 `P0/P1/P2/P3`。现有实现只校验端点之间的相对大小关系，例如 `p1 > p0`、`p2 < p1`，但没有继续扫描 `Pre0-P0`、`P0-P1`、`P1-P2`、`P2-P3` 整段的所有 bar，确认端点确实达到该段应有的最高/最低极值。

这导致一种错误接受：端点之间的数学关系成立，但线段内部存在更低的低点或更高的高点，图上看起来并不是一条合理的折线骨架。问题同时存在于前置下跌先决条件和历史骨架三段，所以这次变更会同时触及 `EvaluatePriorDeclinePrecondition(...)` 与 `BuildHistoricalBackbone(...)` 两条路径。

## Goals / Non-Goals

**Goals:**
- 让 `Pre0-P0`、`P0-P1`、`P1-P2`、`P2-P3` 四段都满足“端点达到整段极值”的要求。
- 保持 cached 历史候选路径和 exact/legacy 对比路径语义一致，避免开启 compare 时出现系统性偏差。
- 明确默认口径为允许并列极值，避免因为完全相等的 high/low 导致候选被过度拒绝。
- 将这条规则定义为检测阶段的结构约束，而不是绘图或交易管理阶段的二次过滤。

**Non-Goals:**
- 本次不改变 `P3-P4` 实时触发段的极值定义。
- 本次不改变 `P5/P6` 激活、止损止盈、观察窗口或共享骨架逻辑。
- 本次不引入“端点必须是唯一极值”的更严格规则。

## Decisions

### 1. 用“段内扫描”验证端点极值，而不是仅凭端点比较

对每一条受约束的线段，都按其起止索引在 `rates[]` 中做一次 inclusive 扫描：
- 上升段要求起点 low 达到段内最低、终点 high 达到段内最高
- 下降段要求起点 high 达到段内最高、终点 low 达到段内最低

这样可以直接消除“端点只是碰巧满足大小关系，但整段内部还有更极端点”的误匹配。

备选方案：
- 仅在候选生成阶段继续加强端点间代数关系。这个方案无法发现线段内部更高/更低的 bar。
- 事后在绘图或日志阶段提示异常。这个方案不能阻止错误交易候选进入后续流程。

### 2. 默认允许并列极值

判定规则采用“达到该段极值”而不是“唯一拥有该段极值”。也就是说：
- 若 `P0.low == min(low over segment)`，则 `P0` 合格
- 若段内其他 bar 也有同样 low/high，不因此否决该段

这样可以避免在真实行情中因为并列 high/low 很常见而过度减少候选。

备选方案：
- 要求唯一极值。这个方案更纯，但会明显降低候选数量，也更容易受报价精度和相同 high/low 噪声影响。

### 3. 先只覆盖 `Pre0-P0` 和历史骨架三段，不把 `P3-P4` 纳入本次 change

`Pre0-P0`、`P0-P1`、`P1-P2`、`P2-P3` 都基于已收盘 bars，适合用同一种扫描口径。`P3-P4` 则依赖实时 tick 与未收盘 bar，语义不同。如果这次把 `P3-P4` 也拉进来，会把“历史结构净化”和“实时触发定义”混成一个 change。

因此本次只修复已经在图上暴露出问题的历史线段与前置线段。

备选方案：
- 同步扩大到 `P3-P4`。优点是更完整，缺点是范围更大，且需要额外界定实时 tick 是否必须是该段最低点。

### 4. 极值校验必须成为骨架接受前的必要条件

`EvaluatePriorDeclinePrecondition(...)` 只有在 `Pre0-P0` 满足极值约束时才能返回通过；`BuildHistoricalBackbone(...)` 只有在 `P0-P1/P1-P2/P2-P3` 都满足极值约束时才能生成有效历史骨架。这样 cached 与 legacy 两条路径都会自然受同一规则约束。

备选方案：
- 让候选先生成，再在更后面统一过滤。可行，但会增加无效候选在缓存中的存在时间，降低搜索效率，也更难保证 compare 输出一致。

## Risks / Trade-offs

- [候选数量下降] → 这是预期结果；通过回测对比确认下降幅度是否合理。
- [历史扫描增加少量计算开销] → 只在候选通过基础代数关系后再做段内扫描，避免对全部索引组合无差别加重负担。
- [cached 与 legacy 行为不一致] → 把极值校验放进共享的骨架/先决条件函数，而不是某一条搜索分支里单独处理。
- [用户对“并列极值是否允许”理解不一致] → 在 spec 和 README 中明确写死默认允许并列极值。

## Migration Plan

1. 在 proposal 对应的 delta spec 中把“段内端点极值”写成检测要求。
2. 实现时先为 `Pre0-P0` 增加扫描校验，再为 `P0-P1/P1-P2/P2-P3` 增加扫描校验。
3. 用 exact compare 和图表回放样本验证：之前不合理但曾被接受的模式应被拒绝。
4. 更新 README 或验收说明，提醒这次会减少一部分原先可匹配的候选。

## Open Questions

- 无。当前边界已经收敛：允许并列极值，且暂不扩展到 `P3-P4`。
