## Context

当前 MT5 P4 模式策略在 `InpAdjustPointMaxSpanKNumber = 5` 时尚可运行，但当该参数提高到 `100` 时，回测耗时显著上升。根因是现有实现将历史结构搜索与实时 P4 触发判断混在一个高频路径中，并且对 P0-P3 使用了近似 `O(span^4)` 的暴力枚举方式，在每次定时扫描时重复执行。

这次变更并不意在改变策略的识别语义，而是要重构搜索路径，使以下两个事实在实现中得到利用：
- P0-P3 只会因新收盘 K 线出现而变化
- P4 是实时触发点，会高频变化

## Goals / Non-Goals

**Goals:**
- 将历史结构搜索与实时 P4 触发评估拆成不同层级，避免在高频轮询中重复重算相同的历史结构。
- 降低 `InpAdjustPointMaxSpanKNumber` 提高到 `100` 后的组合搜索复杂度。
- 保持既有策略规则不变，包括角色化点位取值、CondA-H 语义以及入场/出场逻辑。
- 为后续性能验证提供清晰的缓存与候选点结构。

**Non-Goals:**
- 不改变模式定义、空间变量公式或时间变量公式。
- 不引入并行线程、外部缓存服务或数据库。
- 不改变风控、噪声过滤或持仓管理规则。

## Decisions

### 1. 拆分“历史结构层”和“实时触发层”

实现将被拆为两个处理层：
- 历史结构层：只在检测到新收盘 K 线后更新 P0-P3 候选结构
- 实时触发层：在高频 `OnTimer` 中仅对缓存的历史结构应用当前实时 P4，并执行 CondC、CondD、CondE、CondG、CondH 与下单判断

原因：
- 这是当前性能瓶颈最大的地方。P0-P3 不应在每次 timer 中重复全量枚举。
- 与当前策略“P4 是实时触发点”的语义天然吻合。

备选方案：
- 保留单层流程，仅做局部微优化。否决原因是无法从根本上消除高频重复重算。

### 2. 为每个 symbol 缓存历史候选结构

每个交易品种应维护运行时缓存，至少包括：
- 最近一次参与历史搜索的已收盘 bar 时间
- 候选极值点列表
- 候选历史骨架列表（P0-P3）
- 当前最优或可交易的骨架快照

原因：
- 使“无新 bar”时可以直接跳过历史搜索。
- 让回测和实盘都能复用同一缓存模型。

备选方案：
- 仅缓存单个最优骨架。否决原因是实时 P4 下，不同历史骨架在当前价格下可能出现不同触发结果，保留候选列表更稳妥。

### 3. 先提取候选极值点，再组合历史骨架

历史结构搜索不再直接在全部 bars 上枚举 `i0, i1, i2, i3`。实现应先从已收盘 bars 中提取符合角色语义的候选低点和高点，再在候选极值点之间组合 P0-P3。

原因：
- 候选极值点数量通常远少于全部 bars，能显著降低组合空间。
- 与角色化点位语义一致，避免在明显无效的 bar 上浪费搜索成本。

备选方案：
- 继续枚举全部 bars，只增加剪枝。否决原因是当 span 提升到 `100` 级别时，单纯剪枝不够。

### 4. 在候选骨架组合阶段尽早剪枝

即使在候选极值点层面，也应尽早应用以下轻量条件：
- 点位顺序合法
- 高低点角色关系合法
- 相邻跨度不超过 `InpAdjustPointMaxSpanKNumber`
- 基础空间关系不为负值

原因：
- 尽可能在进入完整变量计算前排除无效组合。

备选方案：
- 所有组合都进入完整变量计算。否决原因是会让缓存更新仍然过重。

### 5. 高频定时器只负责轻量路径

`OnTimer` 的高频路径应仅做：
- 检查是否有新收盘 bar
- 如有则更新缓存
- 使用当前实时价对缓存骨架执行 P4 触发评估
- 管理已有持仓

原因：
- 保留当前高频响应能力，同时避免每次 timer 进入重型搜索。

备选方案：
- 降低 timer 频率。否决原因是这会影响实时触发响应速度，而不是解决算法复杂度问题。

## Risks / Trade-offs

- [缓存与实际 bars 不同步会导致历史结构滞后] -> Mitigation: 用最近收盘 bar 时间作为缓存失效依据，检测到新 bar 后立即刷新。
- [候选极值点预提取可能遗漏合法结构] -> Mitigation: 先定义极值点提取标准，并在验证中与旧实现做结果对照。
- [缓存候选列表会增加内存和状态复杂度] -> Mitigation: 限制缓存窗口并只保留最近 lookback 范围内的数据。
- [性能优化后结果若与旧实现不同，会难以判断是优化问题还是策略问题] -> Mitigation: 增加对照验证，确保在相同输入下匹配结果一致或差异可解释。

## Migration Plan

1. 为每个 symbol 增加历史搜索缓存结构。
2. 从当前单层暴力搜索流程拆出历史结构更新逻辑。
3. 增加候选极值点提取与骨架组合逻辑。
4. 将高频路径改为“缓存更新 + 实时 P4 评估 + 持仓管理”。
5. 用 `InpAdjustPointMaxSpanKNumber = 5` 与 `100` 做性能与结果对照。

Rollback:
- 恢复为当前直接在 `FindLatestPattern()` 中全量枚举历史结构的实现。

## Open Questions

- 是否需要缓存多个历史骨架，还是只缓存最优候选即可满足性能和策略一致性要求。
- 极值点预提取的标准是“局部极值”还是“在跨度窗口内的极值”，这会影响结果一致性和速度。
